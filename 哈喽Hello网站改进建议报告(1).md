## 一、10 项具体改进建议

### 改进 1：匹配倒计时 + 等待期内容运营（P0）

**现状分析：**
匹配在每周一、四 20:00 执行（见 `daily-match.yml`），但 Dashboard 只是文字提示"系统会在每周一、周四晚 8 点为你安排匹配"，**没有任何实时倒计时**。用户在填写 83 题问卷后处于"等待匹配"的真空期，只能刷新页面干等。

**改进方案：**
1. Dashboard 顶部添加实时倒计时 —— "距离下次匹配：1 天 14 小时 23 分钟"
2. 等待期推荐内容卡片：
   - "测测我俩"快速入口（你和谁都可以测）
   - 哈喽墙热门帖子
   - "已有 XX 人在等待本轮匹配"（制造紧迫感）
3. 匹配完成后通过 Email + 站内通知告知（`sendMatchNotification` 函数已存在，确认触发链路完整）

**代码改动：**
- `src/app/dashboard/page.tsx`：新增 Countdown 组件 + 等待期推荐区
- 倒计时基于下次匹配时间的计算（周一/四 20:00 Asia/Shanghai）

**预期效果：** 等待期流失率降低 50%，"测测我俩"引流增加

**执行难度：** ⭐（纯前端，0.5天）

### 改进 2：邀请裂变系统（P0）

**现状分析：**
`src/lib/registrationSources.ts` 已预设 `friend_referral` 渠道，但**没有程序化的邀请码生成和追踪系统**。用户没有一个入口可以查看"我的邀请码"、"我邀请了谁"、获得什么激励。这是增长引擎的核心缺失。

**改进方案：**
1. 每个用户生成唯一 6 位邀请码（用户 ID 短哈希）
2. 注册页面增加"邀请码"可选字段，后台正确关联
3. Dashboard 新增"我的邀请"区域：
   - 复制邀请链接/邀请码
   - 已邀请用户列表（匿名化展示）
   - 邀请进度条
4. **非金钱激励体系**（校园场景不适合现金）：
   - 邀请 3 人 → 解锁专属金句标签"FWD·引路人"
   - 邀请 5 人 → 首页协作者区域展示
   - 邀请 10 人 → 可参与匹配算法优先级微调（等待队列权重提升）

**数据库改动：**
- `User` 表增加 `inviteCode`（唯一）、`invitedByUserId`（可选外键）
- 可选：新增 `InviteRecord` 表或复用 `RegistrationApplication`

**预期效果：**
- 每个活跃用户平均邀请 0.5-1 个新用户（K因子 0.3-0.5）
- 用户增长从纯外部推广变为"外部 + 内生"双引擎

**执行难度：** ⭐⭐⭐（需数据库迁移 + 前后端 + 激励逻辑）

### 改进 3：问卷填写体验优化（P1）

**现状分析：**
83 题问卷是核心竞争力，但也是一大流失点。当前 `src/app/questionnaire/page.tsx`（20,926 字节）一次性展示所有题目，用户可能在中途感到疲惫而放弃。虽已有 `v3` 版问卷完善了问题设计，但填写体验仍有优化空间。

**改进方案：**
1. **分章节展示**：将 83 题分为 6-7 个主题章节（A.对话模式、B1.兴趣投入、B2.兴趣深度、C.三观底色、D.精力节奏、F.生活方式、G.社交风格、E1-E7 恋人维度），每章 8-15 题
2. **进度条**：顶部显示"3/7 章节 · 已完成 35%"
3. **章节过渡动画**：每完成一章，短暂展示该维度的预热文案（"接下来，了解你的社交能量…"）
4. **每页自动保存**：API 已支持 `/api/questionnaire/submit`，改为分页提交，每章完成后自动保存
5. **问卷结果页**：`src/app/questionnaire/result/page.tsx` 当前仅 123 字节（几乎为空），至少显示"你的问卷已提交，正在等待匹配" + 预计下次匹配时间

**代码改动：**
- `src/app/questionnaire/page.tsx`：重构为分章节表单
- `src/app/questionnaire/result/page.tsx`：从 123 字节充实为完整等待页

**预期效果：** 问卷中途退出率降低 40%，完成率提升 30%+

**执行难度：** ⭐⭐⭐（问卷 UI 重构 + 分页逻辑）

### 改进 4：SEO openGraph 元数据补全（P1）

**现状分析：**
`src/app/layout.tsx`（579 字节）已导出基础 `metadata`（title + description），这是好的。但**缺少 `openGraph` 和 `twitter:card` 对象**。这意味着：
- 微信/QQ 分享链接时展示的是空白卡片或仅显示文字
- 没有 OG 图片（1200×630 的社交预览图）
- 没有 sitemap.xml / robots.txt

**改进方案：**
1. 在现有 metadata 基础上补全 openGraph 字段：

```typescript
export const metadata: Metadata = {
  title: '哈喽Hello | 哈尔滨高校同频社交',
  description: '面向哈尔滨高校的同频社交平台...',
  // ↓ 新增以下内容
  openGraph: {
    title: '哈喽Hello | 找到真正同频的人',
    description: '83 题问卷 × 20 类关系维度 = 科学匹配',
    url: 'https://nefudating.top',
    siteName: '哈喽Hello',
    images: [{ url: '/og-image.png', width: 1200, height: 630 }],
  },
  twitter: { card: 'summary_large_image' },
}
```

2. 创建 `/public/og-image.png`（1200×630 推广图，可复用 ShareCard 的 Canvas 渲染逻辑快速生成）
3. 生成 `sitemap.xml` 和 `robots.txt`

**预期效果：**
- 微信/QQ 链接分享显示精美卡片而非空白
- 搜索引擎收录质量提升
- 零成本获取自然搜索流量

**执行难度：** ⭐（半天）

### 改进 5：微信群/公众号承接落地页（P1）

**现状分析：**
首页 Footer 已放置抖音、小红书、X 的社交链接（`src/app/page.tsx` 末尾），但**没有微信群或公众号入口**。推广指南明确指出微信生态（表白墙、班级群、朋友圈）是高校社交的核心渠道。流量来了没有地方沉淀。

**改进方案：**
1. 新建 `/wechat` 页面作为微信生态专属落地页：
   - 公众号二维码 + 关注引导
   - 按学校分类的微信群入口（"NEFU 群 86人"、"哈医大群 42人"）
   - 实时群人数（从数据库动态读取，制造"怕错过"的紧迫感）
2. 首页 Footer 增加"微信群"入口链接
3. 该页面可直接用于微信群/朋友圈推广文案的落地链接

**代码改动：**
- 新建 `src/app/wechat/page.tsx`
- `src/app/page.tsx` Footer 区 +1 行链接

**预期效果：**
- 微信渠道流量有官方沉淀池，避免散落
- 群聊运营数据可视化（人数越多越吸引加入）

**执行难度：** ⭐⭐（新页面）

### 改进 6：首页社交证明增强（P2）

**现状分析：**
首页展示注册人数（394）、匹配对数（243）、学历占比——这些是静态数字。缺少"真实的人正在使用"的动态感。`src/lib/homepageShowcase.ts` 只有 325 字节，仅是重新导出了几个常量。

**改进方案：**
1. **首页增加"最近加入"滚动条**：匿名化展示最近注册用户的学校和年级（"哈尔滨医科大学的硕士刚加入"、"NEFU 的大二同学刚完成问卷"）
2. **匹配成功反馈**：从 `Feedback` 表中筛选正面评价展示（"问卷做完后我等了两天，匹配到了一个和我思维方式惊人一致的人"）
3. **学校排行榜**：各校注册人数 Top 3（"NEFU 186人 · 哈医大 98人 · 哈理工 45人"），制造"我们学校的都在用"的氛围

**代码改动：**
- `src/app/page.tsx` 已有 DB 查询，扩展 `FindMany` 加入新数据源
- `src/lib/homepageShowcase.ts`：从 325B 的纯常量扩展为动态数据获取

**预期效果：** 首页访问→注册转化率提升 20-40%

**执行难度：** ⭐（简单 DB 查询）

### 改进 7：哈喽墙内容运营化（P2）

**现状分析：**
`/community` 页面已实现发帖和评论功能。但作为公开页面在导航栏展示，可以更好地充当**内容营销着陆页**。

**改进方案：**
1. 社区首页由管理员精选 3-5 条高质量帖子置顶
2. 帖子分类标签：#匹配故事 #校园安利 #灵魂提问 #脱单攻略
3. 未登录用户可以浏览，发帖/评论时引导注册（"登录后参与讨论"）
4. **关键价值：** 哈喽墙的 UGC 内容可以直接作为小红书/抖音的内容素材来源——运营人员写推广文案时有真实素材可用

**预期效果：**
- 哈喽墙成为自然 SEO 内容来源
- 为社交媒体运营提供 UGC 素材
- 增加网站粘性（用户来了不只匹配，还能逛墙）

**执行难度：** ⭐⭐

### 改进 8：Dashboard 匹配等待期状态细化（P2）

**现状分析：**
Dashboard 已有状态管理（待补充资料 / 待问卷 / 待匹配 / 已匹配 / 本轮落选），这是很好的基础。但缺少对"正在等待本轮匹配"的具体引导。

**改进方案：**
1. "正在等待"状态下，除了倒计时（改进 1），增加可操作内容：
   - 修改匹配偏好（性别、目的）的快捷入口
   - "刷新你的问卷" 引导（重新审视 83 题答案）
   - 分享邀请链接给朋友一起等
2. "本轮落选"状态下，增加安抚信息：
   - "落选意味着系统暂时没有找到足够契合的匹配，不代表你不够好"
   - 等待队列优先级说明（"下一轮你将拥有更高优先级"）
   - 推荐去哈喽墙看看

**预期效果：** 减少 "为什么我还没匹配上" 的焦虑和弃用率

**执行难度：** ⭐⭐

### 改进 9：Aha Moment 检测机制（P2）

**现状分析：**
推广指南第一章强调："在明确产品对于市场是'不可或缺'还是'有没有都无所谓'之前，都不应制订过于雄心勃勃的增长计划。" 当前没有任何机制检测用户是否体验到了"啊哈时刻"。

**改进方案：**
1. 匹配结果查看后 48 小时，Dashboard 底部出现 NPS 调查："你会把哈喽推荐给朋友吗？" （0-10分）
2. `/api/auth/me` 扩展返回字段，携带 `npsScore` 和 `npsSubmittedAt`
3. 后端统计关键行为转化率：
   - 注册 → 完成资料 → 完成问卷 → 查看匹配 → 回复/互动
   - 每个阶段的流失率

**代码改动：**
- `src/app/dashboard/page.tsx`：NPS 组件
- `src/app/api/auth/me/route.ts`：扩展返回字段
- 新建 `src/app/api/feedback/nps/route.ts`

**预期效果：**
- 获取产品核心指标数据，指导后续优化方向
- 发现可能导致用户流失的产品缺陷（如问卷太长、匹配太慢、卡片不够好看等）

**执行难度：** ⭐⭐

### 改进 10：微信小程序版（P3，长期）

**现状分析：**
当前是纯 Web 应用，托管在 Vercel。中国高校场景中微信小程序有天然优势：免安装、扫码即用、朋友圈分享更自然。

**改进方案：**
使用 Taro 或 uni-app 封装核心功能（注册、问卷、匹配结果查看）为微信小程序。Web 版保留完整功能。小程序聚焦"轻量体验 + 引流 Web"。

**注意事项：**
- 社交类小程序审核较严，需要企业资质
- 建议等用户量到 1000+ 后再启动

**执行难度：** ⭐⭐⭐⭐⭐（新项目开发 + 审核周期）
